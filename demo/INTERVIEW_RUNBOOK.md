## Post-demo Q&A guide

### Q1. Why Qwen2.5-7B-Instruct over a frontier LLM SaaS?

Three reasons. (1) **Portfolio-aligned with Sonatus's edge-AI thesis.** Sonatus AI Director ships both SLMs and LLMs because the vehicle telemetry boundary is the right trust boundary for an open-weight model — that's Principle XIII, load-bearing claim, not cost optimization. (2) **Reproducibility.** Qwen2.5-7B-Instruct pinned to revision SHA `a09a35458c702b33eeacc393d103063234e8bc28` — Hugging Face commit, not a tag. Apache-2.0 license. Weight content is verifiable by SHA-256 at container start; runtime image digest-pinned in the Dockerfile. A SaaS LLM is a black-box dependency for an OEM-customer audit trail. (3) **Schema-constrained decoding.** outlines is bound to the `CollectionPolicySpec` Pydantic v2 schema at decode time; the SLM cannot emit a non-schema-conforming response by construction. Frontier LLM APIs with response_format support are catching up, but the constrained-decoding library against an open-weight runtime is the lower-trust path. ADR-0002 + ADR-0003.

### Q2. How do you handle RLS connection pooling?

`Database.acquire(tenant_id)` opens an asyncpg pool connection, executes `SET LOCAL app.tenant_id = $1; SET LOCAL ROLE collectmind_tenant;` inside the transaction, then yields. `collectmind_tenant` is non-BYPASSRLS — even with the application's connection credentials, the role drop forces every SELECT against tenant-scoped tables to traverse the RESTRICTIVE policy. SET LOCAL means the role-drop reverts automatically on transaction close; the pooled connection returns to the role the application connected as (`collectmind`, which IS BYPASSRLS for migrations and break-glass). This is the Phase 9.b worked example — RESTRICTIVE-only RLS without a PERMISSIVE baseline returned zero rows for everyone; the fix combined a `PERMISSIVE USING (true)` baseline policy with the per-tenant RESTRICTIVE filter so the Postgres policy combiner has a permissive input to AND against. ADR-0007 Part 3.

### Q3. What's the cost of break-glass?

Three costs. (1) **Audit visibility.** Every invocation writes a `kind=break_glass` row keyed by `(operator_principal_subject, tenant_scope, reason_code, correlation_id)` — fully attributable, retroactively queryable. (2) **Alerting cost.** `BreakGlassInvoked` pages on every single invocation per `(operator, reason_code)`. `BreakGlassBurstInvocation` is critical-tier on > 0.05/s for 5 min per operator. The two split is deliberate per the Phase 13 review — a legitimate burst (legal-hold retrieval) and an abuse burst (operator-key compromise) have the same trigger but different responses, so two playbooks. (3) **Schema cost.** The reason_code enumeration is closed — `incident_response`, `legal_hold`, `regulator_request`, `support_escalation`, `operator_self_audit`. Extending it requires an ADR amendment to ADR-0007. Operators can't invent novel reasons mid-incident.

### Q4. How was the dashboard-lag SLO measured?

T136 measurement methodology, recorded inline in `observability/runbooks/slo-006-dashboard-lag.md`. Five `POST /findings` publications spaced 12 seconds apart. For each: timestamp at publish, poll Prometheus's `collectmind_diagnostic_findings_received_total` until the rate sample reflects the new event, record the wall-clock delta. Max across the five: 2.11 s. Mean: 1.98 s. Against a 10-second SC-006 budget — about five-times headroom. The Compose `scrape_interval` is 2 s — a hard floor on freshness. Production scrape intervals pinned at the same 2 s in Terraform unless an ADR records a deviation. The methodology is reproducible; running `python scripts/measure_dashboard_lag.py` against a warm stack reproduces the numbers within ±20 % run-to-run.

2. **SC-009 rolling-5-PR wall-clock aggregator** → gated to the first PR-tier run trending toward 18 minutes. Inaugural run was 11 min 02 s; aggregator not needed at the inaugural measurement.

### Q6. How does the test-first cadence actually work?

Phase 3 closure is the canonical worked example. The user reviewed all 17 US1 test files (T048–T064) end-to-end before any implementation began (T065+). Test files committed in the red phase at `9c4bd7d`; implementation followed at `b9fddc8` after the test design was approved. Reviewing tests-only — without implementation in scope — means the reviewer is reasoning about the contract, not the code. The pattern surfaced test-design defects (e.g., the tautological inclusive-on-confirm assertion in T056) before any implementation locked in a wrong shape. Phase 9.b under feature 002 followed the same shape: `4d18a22` red, `5429689` impl. Principle IV's "tests are written before or alongside implementation, never after the fact" is the constitutional anchor; the commit log is the audit trail.

### Q7. What's the difference between recorded mode and live mode in the demo?

Same UI, same TypeScript code paths, different fetch implementation. Recorded mode resolves a `(method, path, principal, body)` tuple against `public/recordings/index.json` which Vite bundles into the JS — no network at runtime. Live mode calls `window.fetch` with the principal's bearer token from `.env.local` (gitignored, never shipped). The deployed Vercel/Pages build is recorded-only because the deployed UI can't reach localhost — Q2 decision in the build prompt. Live mode is for the local dev loop where the Compose stack is in front of the developer. Same `ApiError` surface, same retry-after parsing, same correlation_id extraction — the UI doesn't know which mode it's in.

### Q8. How is the demo UI's own coverage measured?

Vitest + RTL + v8 coverage. 75 tests across 10 files. Thresholds in `vitest.config.ts`: `lines ≥ 85, statements ≥ 85, functions ≥ 85, branches ≥ 80`. Current numbers: 95.99 / 95.99 / 93.68 / 89.29 — well over the floor. Same Principle IV 85 % gate as the backend, measured separately. The backend coverage is across `src/collectmind/` only; the demo coverage is across `demo/src/` only. `make demo-test` runs the full suite locally; `npm run test` produces the HTML report at `demo/coverage/`.

### Q9. What's the deal with the SLM container's network isolation?

Terraform security group `aws_security_group.slm` at `infra/terraform/networking/main.tf` allows egress only on ports 4317 and 4318 to the OTLP collector inside the VPC. No 443 outbound. No 5432 to the registry. No 9092 to Kafka. No 53 to public DNS. The SLM container resolves the OTLP collector by VPC-internal hostname only. This is Principle XIII's "isolated container with no outbound network except to the configured observability endpoints" enforced at the IaC level. Locally in Compose the same isolation is approximated by a dedicated network and an egress-block rule. If the SLM ever generated text suggesting an exfiltration vector — say, an HTTP URL in a hallucinated trigger condition — the network would block the call before it left the container.

### Q10. What would you do differently?

Two answers depending on horizon. **Tactically, in the next feature**: front-load the inaugural-CI-against-renamed-remote work. PR #1 took 13 fixup commits to land the first green CI run — semgrep dep conflict, terraform fmt drift, gitleaks-action token requirement, ruff drift, supply-chain CVE backlog. Those are inaugural-PR-fingerprint costs; future feature-reopen cycles inherit a playbook that runs the full PR-tier CI on a stale-tooling-aware staging branch before opening the feature PR. **Strategically, in the architecture**: tighten the SLM upgrade path. ADR-0002's upgrade procedure says "ADR records prior + new revision SHAs, eval-suite delta, rollback procedure" — but the runtime rollback path isn't a one-command operation today; it requires re-pulling weights, re-running the readiness check, and re-validating the contract corpus. Worth landing a `slm_upgrade.py` automation that drives the rollback under the same gating that drives the upgrade.

### Q11. The fixtures look real. Are they captured or hand-written?

Hand-written, but every field traces to a documented constant in the repo. Schema-correct per the OpenAPI v1.1.0 contracts. SLM repo + revision SHA from ADR-0002; decoding seed from `_DEFAULT_DECODING_SEED = 0xC0FFEE` in `policy_generator.py`; prompt template version `1.0.0` from `graph/session.py`; tenant ids and vehicle ids from the feature-002 quickstart; VSS leaf names from the Phase-3 closure fix. `demo/scripts/record_fixtures.sh` is the canonical re-capture path against a running Compose stack — running it replaces the synthesized pack with a real-run capture. The DECISIONS.md entry at 2026-05-14 names this explicitly so the next reviewer knows the difference between "captured" and "schema-derived." This isn't a Principle XI no-fabrication violation — the SLO numbers on `/slo` all source to runbook artifacts; the fixtures are demo *shape*, not measurement claims.

### Q12. What's the demo UI deployed on, and why not Vercel as you wrote?

GitHub Pages at `arunveligatla99.github.io/snts_collectmind`. The session that built the demo had no Vercel auth context — `vercel whoami` returned 401, no `VERCEL_TOKEN` in env, `vercel login` is an interactive web-OAuth flow that headless sessions can't drive. GitHub Pages was reachable through the existing `gh auth status` keyring entry, so the deploy was one push to `gh-pages` + one `gh api -X POST repos/.../pages` call. The build artifact itself is host-neutral: Vite reads `VITE_BASE_PATH` via `loadEnv` from `.env.production`, so the same `npm run build` serves Pages at `/snts_collectmind/` or Vercel at root by flipping that one value. React Router's `basename` and the static link helpers in `Landing.tsx` both pick up `import.meta.env.BASE_URL`. `vercel.json` stays in place; future Vercel deploy is `cd demo && vercel --prod` with the user's credentials.

### Q13. How are the contracts tested?

Three layers. (1) **schemathesis fuzz** against the OpenAPI 3.1 documents at `contracts/openapi/orchestration-api.v1.yaml`, `query-api.v1.yaml`, `audit-admin.v1.yaml`. The fuzzer generates payloads that satisfy the schema, sends them, asserts that the response shape matches the documented response. The fuzz caught the path-parameter `%00`-byte 500 during Phase 3 closure — `_ensure_safe` validator now lands non-printable bytes on 404 instead. (2) **AsyncAPI conformance** for each of the four Kafka topics (`diagnostic-findings`, `policy-events`, `vehicle-telemetry`, `policy-outcomes`); the conformance harness publishes a synthetic event and asserts the consumer's parsed shape matches the AsyncAPI 3.0 document. (3) **PolicyGeneratorClient contract test** at `tests/contract/test_slm_client_contract.py` runs the same scenario against all four implementations (vLLM, llama.cpp, FingerprintStub, DevDefault). Real-SLM contract test runs under `temperature=0` + fixed seed per Principle XIV. The four implementations passing the same contract is what lets a future cloud-LLM adapter ship without changing the graph.

### Q14. How is OTel propagation wired across the LangGraph nodes?

OTel SDK at the orchestration-api entry point sets up a tracer provider exporting to OTLP. The four LangGraph nodes (Orchestrator, Policy Generator, Validator, Deployer) each open a child span with the node name + the `correlation_id` as a span attribute. The Kafka producer/consumer pair propagates trace context via W3C trace-context headers in the Kafka headers field — the consumer extracts on `poll()` and resumes the span. The SLM client's HTTP call to vLLM goes through `httpx` with the OTel instrumentation, so the SLM call shows up as a nested span under the Policy Generator node. The dashboard at `infra/compose/grafana/dashboards/collectmind.json` has a trace-view panel that links from any audit row's `correlation_id` directly to the trace. Tempo (or Jaeger in CPU profile) is the trace backend. The propagation contract is documented in `docs/architecture/observability.md`.

### Q15. What's your IaC strategy?

Terraform per module under `infra/terraform/`. Modules: `networking` (VPC, subnets, the SLM-isolation security group), `secrets` (AWS Secrets Manager entries + scope-tight IAM policies; ECS execution role intentionally does NOT have Secrets Manager read — Phase 5 deliberate decision per least-privilege), `compute` (ECS-on-EC2 with g5/g6 GPU node group per ADR-0005), `db` (RDS Postgres with TimescaleDB, parameter group with `shared_preload_libraries='timescaledb'`), `observability` (ADOT collector, AMP workspace, Grafana). Plan-only in PR-tier CI; apply gated to workflow_dispatch. One Phase-7 follow-up: T244 — Terraform `null_resource` that invokes the migration runner. Three options on the table per the readiness review (one-shot `local-exec` vs sidecar-init-container vs ECS task with `dependsOn`); the choice is bundled with the cloud-deploy hardening feature, not made mid-implementation.

### Q16. What would you do differently in feature 003?

Three specific calls. (1) **Run the inaugural-CI sweep on a staging branch before the feature PR.** Feature 002's PR #2 surfaced three feature-002-native CI issues (postgres-init `.up`/`.down` ordering, unit-tier coverage drop, rate-limit contract tests under CI's docker-compose network namespace) — all real, all fixable, but all surfaced under PR-tier CI which raised the wall-clock cost. Future feature opens with a "feature ready for CI" branch that gets a CI dry-run before the PR opens. (2) **Front-load the supply-chain refresh.** Trivy + pip-audit + Syft are currently informational; that's a deliberate Phase-7 deferral but it costs visibility on every PR. Future feature opens with a freshness sweep on the deps that gate it. (3) **Add a feature-flag layer.** Feature 002's hot-store key migration used an env-var-gated dual-read window with a Fatal deadline — that worked, but a real feature-flag service with audited rollouts would let production traffic shape the cutover instead of an operator timing it. Out of scope for an interview demo, in scope for a real OEM deployment.

## Failure recovery — extended

### (e) Network drops mid-demo

Symptoms: deployed URL stops loading; mid-click panel freezes; spinning cursor on every action.

Recovery:
1. Don't panic; don't repeatedly click — that queues actions and confuses the UI on reconnect.
2. Verbal: "Network hiccup — let me re-share when it's back." Stop screen-sharing for 15 seconds; check connection on the side; restart sharing.
3. If the network stays down past 60 seconds, switch to the deck. Slides 4 (architecture) + 8 (measured SLOs) + 9 (audit FR-017a) carry the same story without the live UI.

### (f) Screen-share goes black or shows the wrong window

Symptoms: interviewer says "I can't see anything" or "you're sharing your email."

Recovery:
1. Stop sharing immediately. Re-share — pick the correct browser window, not the whole screen if you can avoid it.
2. Verbal: "One second — let me re-share the browser window."
3. If you accidentally shared something with sensitive content (email, Slack, tokens), apologize once, re-share the correct window, move on. Don't dwell.

### (g) Demo URL serves stale content after a code change

Symptoms: a panel renders differently than expected; the snapshot SHA in the header doesn't match `arun/demo-ui` HEAD; an ADR's content seems out of date.

Recovery:
1. Pre-demo only — never debug this mid-call.
2. `git push origin arun/demo-ui` to push the latest. Then `npm run build` in `demo/`. Then re-stage `dist/` to `/tmp/gh-pages-deploy` and force-push to `gh-pages`. The GitHub Pages CDN takes 60–90 seconds to roll over.
3. If the snapshot SHA still doesn't match after the rollover, the bundler ran against an older repo state. Re-run `node demo/scripts/bundle_content.mjs` from a clean checkout.

### (h) Interviewer says "the audit chain looks fake to me"

Recovery:
1. Don't get defensive. The honest answer is that the deployed UI runs in recorded mode and the fixtures are schema-derived from documented project constants (ADR-0002 SHA, `_DEFAULT_DECODING_SEED`, the feature-002 quickstart's tenant ids). DECISIONS.md 2026-05-14 names this explicitly.
2. Offer the live-mode toggle if Compose is up. Switch, repost, watch the chain materialize from real Postgres rows.
3. If Compose isn't up, offer to walk the code. Open `src/collectmind/registry/audit.py:write()` in the third browser tab. Point at the FR-017a enforcement (`raises ValueError if any FR-017a field is missing for kind=generated`). The contract is in the code, regardless of which mode you're rendering.
4. Verbal: "Recorded mode is the deployed default — Q2 build-prompt decision. Recapture runs `bash demo/scripts/record_fixtures.sh` against the live stack. The fixtures match the contracts exactly; the demo is showing the contract, not validating it."

### (i) Interviewer asks a question I don't know the answer to

Recovery:
1. Don't bluff. "I haven't measured that" or "I haven't shipped that yet" is a strictly better answer than a fabricated number.
2. Anchor the question to the closest thing you have. If they ask about p99 latency under sustained 1000 ev/s load, the truthful answer is: "Locally I measured p50 at 50 ms over a 60-second smoke load with 280 requests and 0 failures (T134); p99 at sustained 1000 ev/s/tenant is gated to workflow_dispatch SC-002 which hasn't been run yet — that's the same gating pattern as ADR-0002's eval baseline."
3. Offer to follow up: "I'll measure that and send a summary in the post-call thread."
4. Move on. Don't dwell on the unknown.
